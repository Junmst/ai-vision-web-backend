# Web 应用课程综合设计说明书

## 一、项目背景

本项目名称为 **AI Vision Web 智能图片分析平台**。项目面向日常图片内容理解与 AI 辅助分析场景，目标是实现一个可运行、可部署、可演示的全栈 Web 应用，使用户能够通过浏览器完成账号注册、登录、图片上传、图片管理、AI 图像分析、分析历史查看以及个性化 AI 接口配置等操作。

在当前 AI 大模型快速发展的背景下，图像理解类能力逐渐从研究走向实际应用。传统图片管理系统通常只能完成图片上传、浏览、分类等基础操作，而本项目在此基础上引入了 OpenAI 兼容大模型接口，使系统能够进一步对图片进行智能描述、关键词提取、场景识别以及 Prompt 反推，从而提升系统的智能化水平与实际应用价值。

本项目同时满足课程任务书中对于“前端 + 后端 + 数据库”完整三层架构的要求，属于一个典型的 Web 全栈应用。前端负责用户交互界面和页面逻辑，后端负责接口服务、业务处理与安全控制，数据库负责用户、图片、分析记录和 AI 配置等核心数据的持久化存储。

## 二、需求分析与可行性分析

### 2.1 功能需求分析

根据课程设计任务目标和实际应用场景，本系统主要需要实现以下功能：

1. **用户管理功能**  
   用户可以完成注册、登录、身份验证和退出登录。系统需要对不同用户的数据进行隔离，确保每位用户只能访问自己的图片、分析记录和 AI 配置。

2. **图片上传与管理功能**  
   用户可以从本地设备上传 JPG、PNG、GIF、WEBP 等格式的图片，系统需要对图片的类型和大小进行校验，并将图片保存到服务器本地目录，同时记录图片元数据到数据库。

3. **AI 图片分析功能**  
   用户上传图片后，可以点击“分析”按钮调用大模型接口，对图片内容进行智能分析。系统支持多种分析模式，包括图片描述、标签提取、场景识别和 Prompt 反推。

4. **分析结果管理功能**  
   系统需要记录每一次分析结果，包括使用的模型、分析类型、耗时和返回内容，便于用户查看历史分析记录。

5. **用户自定义 AI 配置功能**  
   用户可以自行配置第三方 AI 平台的 `base_url`、`api_key`、模型名称、鉴权方式等内容，方便系统适配不同的 OpenAI 兼容平台。

6. **系统部署与运行功能**  
   系统应支持在服务器上部署运行，前端通过 Nginx 提供静态资源访问，后端通过 FastAPI 提供 API 服务，并支持公网访问与答辩演示。

### 2.2 非功能需求分析

除核心业务功能外，系统还需要满足以下非功能需求：

- **可用性**：界面操作简单清晰，用户可以快速完成上传、分析和查看结果等流程。
- **安全性**：用户密码不能明文保存，必须加密存储；接口需要进行身份认证；用户数据需要隔离。
- **可扩展性**：后端需要便于后续新增模型平台或新的分析模式。
- **可维护性**：系统架构应清晰分层，代码应具备较好的可读性和可解释性，便于答辩与后续维护。
- **性能与稳定性**：上传、查询、AI 请求等操作需要具备基本稳定性，支持图片大小上限控制和服务异常处理。

### 2.3 可行性分析

#### （1）技术可行性

本项目采用 Vue 3 作为前端框架，FastAPI 作为后端框架，SQLite 作为数据库，整体技术方案成熟且实现成本低。FastAPI 拥有较好的异步支持和自动接口文档功能，适合课程设计快速搭建后端；SQLite 部署简单，适合中小规模课程项目；Nginx 可完成前后端分离部署。

#### （2）实现可行性

系统的各项功能边界明确，模块划分清楚，开发难度适中。前端负责页面与请求交互，后端负责业务接口与 AI 调用，数据库负责存储。由于本项目允许使用 AI 辅助开发，因此在开发效率上具有优势，但小组成员仍需理解全部代码逻辑。

#### （3）经济可行性

项目开发环境成本较低，使用的 Python、Vue、SQLite、Nginx 等均为开源工具。AI 能力通过用户可配置 API 接口实现，系统本身无需绑定特定商业服务。

#### （4）运行可行性

系统当前已可以运行在服务器环境中，具备完整的访问入口、图片上传能力、AI 分析能力和历史记录能力，满足课程答辩演示需求。

## 三、软件架构设计

### 3.1 总体架构

本项目采用典型的 **前后端分离三层架构**：

1. **表示层（前端）**：基于 Vue 3 构建用户交互界面。
2. **业务层（后端）**：基于 FastAPI 提供 RESTful API，处理业务逻辑。
3. **数据层（数据库）**：基于 SQLite 存储用户、图片、分析记录和 AI 配置等数据。

系统部署时，前端构建后的静态文件放置在 Nginx 目录中，由 Nginx 直接提供网页资源；所有 `/api/` 请求由 Nginx 反向代理到 FastAPI 服务；图片文件存储在后端数据目录，数据库保存对应元数据。

### 3.2 后端分层设计

后端采用模块化分层结构：

- `main.py`：应用入口，负责创建 FastAPI 实例、注册路由、挂载静态目录。
- `config.py`：配置层，负责统一读取 `.env` 中的环境变量。
- `database.py`：数据库层，负责创建 SQLAlchemy 引擎、会话工厂和依赖注入。
- `models.py`：ORM 模型层，定义数据表结构与实体关系。
- `schemas.py`：数据校验层，定义请求与响应模型。
- `auth.py`：认证安全层，负责密码哈希、JWT Token 签发和身份校验。
- `routers/`：路由层，负责接收前端请求和分发业务逻辑。
- `services/`：服务层，负责 AI 分析调度与外部平台调用。

这种结构使得后端各模块职责明确，便于后续扩展与维护。

### 3.3 前后端通信方式

前端通过 HTTP 请求调用后端 API，采用 JSON 或表单数据格式完成交互。登录成功后，前端将后端返回的 JWT 保存下来，后续请求统一通过 `Authorization: Bearer token` 的方式访问受保护接口。

## 四、功能设计

### 4.1 注册功能

当用户在注册页面输入用户名和密码后，前端会调用 `POST /api/auth/register` 接口。后端收到请求后，首先检查用户名是否已存在；如果不存在，则使用 `bcrypt` 对密码进行加盐哈希，再将用户信息保存到数据库中的 `users` 表。

这一过程实现了账号创建功能，并保证密码不会以明文形式保存。

### 4.2 登录功能

登录时，前端将用户名和密码以表单形式提交到 `POST /api/auth/login`。后端首先根据用户名查询用户记录，然后使用 `bcrypt` 验证密码是否正确。如果验证成功，则通过 `python-jose` 生成 JWT Token 并返回给前端。

之后，前端在访问图片上传、图片列表、分析记录、AI 配置等接口时，都会携带该 Token，后端通过依赖注入方式校验当前用户身份。

### 4.3 图片上传功能

用户选择本地图片后，前端通过 `multipart/form-data` 上传到 `POST /api/images`。后端执行如下逻辑：

1. 校验文件类型是否为 JPG、PNG、GIF、WEBP；
2. 校验文件大小是否超出限制；
3. 生成唯一文件名，避免重名冲突；
4. 将图片保存到服务器本地 `data/uploads/` 目录；
5. 在 `images` 表中记录文件名、文件路径、大小、所属用户和上传时间。

这样实现了“文件存磁盘、信息存数据库”的设计方式，便于管理和查询。

### 4.4 图片列表与历史记录功能

用户进入首页或历史记录页面时，前端会向后端请求图片列表或分析记录列表。后端查询时会使用 `user_id` 作为过滤条件，只返回当前登录用户自己的数据，从而保证多用户数据隔离。

### 4.5 AI 图片分析功能

这是本项目的核心功能。当用户点击“分析”按钮时，后端主要执行以下流程：

1. 接收前端传来的 `image_id` 和 `analysis_type`；
2. 校验该图片是否属于当前用户；
3. 根据数据库中的图片路径读取图片文件；
4. 将图片转成 Base64 编码，以适配 OpenAI 兼容图像接口；
5. 查询用户 AI 配置，选择默认配置或指定配置；
6. 根据配置中的 `auth_mode` 决定请求头鉴权方式；
7. 使用 `httpx.AsyncClient` 调用外部 AI 接口；
8. 将模型返回结果解析成统一结构；
9. 把分析结果写入 `analysis_results` 表，并记录耗时；
10. 将结果返回前端显示。

本系统支持以下几类分析：

- 图片整体描述
- 图片标签提取
- 场景识别
- Prompt 反推
- 综合分析

### 4.6 AI 配置管理功能

用户可以在 AI 配置页面自行填写大模型接口地址、API Key、模型名称和鉴权方式。后端通过 `ai_configs` 表保存这些配置。若用户设置某一项为默认配置，系统会自动取消其他默认项，从而确保每个用户只有一个默认配置。

这一功能增强了系统的通用性，使项目不依赖某一个固定平台，而是可以适配多个 OpenAI 兼容服务。

### 4.7 删除功能

无论是删除图片还是删除 AI 配置，后端都会先验证该资源是否属于当前用户，再执行删除操作。这样既保证数据安全，也防止越权访问。

## 五、数据库设计

本项目数据库采用 SQLite，包含以下 4 张核心数据表：

### 5.1 用户表 `users`

用于保存用户账号信息，字段主要包括：

- `id`：主键
- `username`：用户名
- `password_hash`：密码哈希值
- `created_at`：创建时间

### 5.2 图片表 `images`

用于保存用户上传的图片信息，字段主要包括：

- `id`：主键
- `user_id`：所属用户外键
- `filename`：原始文件名
- `file_path`：服务器保存路径
- `file_size`：文件大小
- `created_at`：上传时间

### 5.3 分析结果表 `analysis_results`

用于保存 AI 分析记录，字段主要包括：

- `id`：主键
- `user_id`：所属用户
- `image_id`：对应图片
- `provider`：所使用的平台类型
- `model_name`：模型名称
- `analysis_type`：分析类型
- `result_json`：分析结果
- `latency`：接口耗时
- `created_at`：分析时间

### 5.4 AI 配置表 `ai_configs`

用于保存用户自己的 AI 接口配置，字段主要包括：

- `id`：主键
- `user_id`：所属用户
- `config_name`：配置名称
- `provider`：平台类型
- `base_url`：接口地址
- `api_key_encrypted`：API Key
- `auth_mode`：鉴权方式
- `auth_header`：请求头名称
- `model_name`：模型名称
- `is_default`：是否为默认配置
- `created_at`：创建时间

### 5.5 表关系设计

- 一个用户可以拥有多张图片；
- 一个用户可以拥有多条分析记录；
- 一个用户可以拥有多个 AI 配置；
- 一张图片可以对应多次分析结果。

这一设计既满足业务逻辑，也有利于后续系统扩展。

## 六、关键技术路线与选型理由

### 6.1 前端技术选型

前端采用 Vue 3。原因在于 Vue 3 组件化开发方便，适合构建结构清晰的单页应用；同时其生态成熟、学习成本相对较低，适合作为课程设计项目的前端实现方案。

### 6.2 后端技术选型

后端采用 FastAPI。原因在于：

- 接口定义清晰，开发效率高；
- 原生支持异步；
- 自动生成 Swagger 文档，便于测试和答辩演示；
- 与 Pydantic、SQLAlchemy 集成度高。

### 6.3 数据库选型

数据库采用 SQLite。原因在于：

- 部署成本低，无需单独安装数据库服务；
- 非常适合课程设计和小型项目；
- 配置简单，便于在答辩环境快速复现。

### 6.4 AI 接口调用方案

系统没有直接绑定某一家大模型厂商，而是选择 OpenAI 兼容接口方式。这样可以兼容硅基流动、DeepSeek、智谱等不同服务平台，提升系统通用性。后端增加了对 `Bearer Token` 和 `API-Key` 两种鉴权方式的支持，使得用户可以自己填写不同厂商的接口配置。

### 6.5 反向代理与部署方案

系统采用 Nginx 作为统一入口：

- 前端静态页面由 Nginx 提供访问；
- `/api/` 请求转发到 FastAPI；
- `/uploads/` 路径访问上传图片；
- 通过 `client_max_body_size` 控制最大上传大小。

这种部署方式清晰稳定，符合常见 Web 项目工程实践。

## 七、对社会、经济、法律、安全、健康等因素的考量

### 7.1 数据安全与账户安全

本项目对用户密码采用 `bcrypt` 哈希加密存储，避免明文泄露风险。系统通过 JWT 校验登录身份，防止未登录用户访问受保护资源。数据库查询时严格使用 `user_id` 进行数据过滤，保证不同用户间的数据隔离。

### 7.2 隐私与合规问题

本系统允许用户上传图片并调用第三方 AI 平台进行分析，因此需要提醒用户避免上传包含个人敏感隐私的信息图片。在系统设计层面，AI 接口配置为用户自行提供，项目本身不强绑定特定外部厂商账号。

### 7.3 内容安全与输出合理性

AI 分析结果由第三方模型生成，可能存在不准确、偏差或不稳定的问题。因此系统在设计上采用“分析结果展示”而非“自动决策执行”，用户可以将结果作为辅助参考，而不能完全替代人工判断。

### 7.4 经济性考量

项目整体使用开源框架和轻量数据库，运行成本低。AI 模型调用成本由用户自行配置的平台决定，提高了系统的灵活性和可控性。

## 九、系统运行与功能流程说明

### 9.1 用户注册流程

用户在注册页面输入用户名和密码后，系统通过后端接口创建账号。后端先检查重名情况，再对密码进行哈希处理并保存到数据库。

### 9.2 用户登录流程

用户登录时，系统验证用户名与密码，成功后生成 Token 返回前端。之后所有业务请求都依赖该 Token 进行认证。

### 9.3 图片上传流程

用户选择图片并点击上传后，后端会验证图片类型和大小，将图片保存到磁盘目录，并在数据库中记录图片信息。

### 9.4 图片分析流程

用户点击分析按钮后，后端读取图片文件，将其编码后发送给大模型接口，再将返回结果解析并记录入库，最后返回前端展示。

### 9.5 AI 配置流程

用户在配置页保存自己的 API Key、模型和接口地址后，后端会将这些信息与用户账号关联保存，后续分析优先使用该配置。
