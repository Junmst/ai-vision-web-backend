# AI Vision Web — 后端架构说明

> **答辩用技术文档** | 更新时间：2026-07-14

---

## 一、项目简介

AI Vision Web 是一个基于 Web 的 AI 图片分析平台。用户上传图片后，系统调用 OpenAI 兼容的大模型 API，自动返回图片描述、关键词标签、场景识别、Prompt 反推等结构化分析结果。支持用户自定义 API 配置，适配多种 AI 平台。

## 二、技术栈

| 层级 | 技术 | 作用 |
|------|------|------|
| **Web 框架** | FastAPI 0.115 (Python) | 异步高性能 API 框架，自动生成 Swagger 文档 |
| **ASGI 服务器** | Uvicorn 0.34 | 驱动 FastAPI 运行，监听 8000 端口 |

| **数据库** | SQLite | 轻量嵌入式数据库，零配置部署 |
| **认证** | JWT + bcrypt 4.2 | 无状态 Token 鉴权，密码加盐哈希 |
| **HTTP 客户端** | httpx 0.28 (async) | 异步调用外部 AI 平台 API |
| **反向代理** | Nginx | 静态文件托管 + API 反向代理 + 上传限制 |
| **进程守护** | systemd | 服务自启、崩溃自动重启、日志管理 |

## 六、核心业务流程

### 6.1 AI 分析完整流程

```
POST /api/analysis  { image_id: 1, analysis_type: "all" }
         │
         ▼
  ┌─ ai_service.run_analysis() ───────────────────────────────────┐
  │                                                                │
  │  Step 1: 读取图片文件                                          │
  │    image.file_path → open() → base64 编码                      │
  │                                                                │
  │  Step 2: 获取 AI 配置（优先级从高到低）                          │
  │    ① 请求参数指定 ai_config_id → 查 ai_configs 表              │
  │    ② 未指定 → 查用户 is_default=1 的配置                       │
  │    ③ 无配置 → 回退 .env 系统默认配置                            │
  │                                                                │
  │  Step 3: 选择 Provider + 鉴权方式                              │
  │    auth_mode = "bearer"  → Authorization: Bearer {key}        │
  │    auth_mode = "api-key" → {自定义Header}: {key}               │
  │                                                                │
  │  Step 4: 构建 Prompt + 调用 AI                                 │
  │    build_system_prompt(analysis_type) → OpenAI 多模态消息格式   │
  │    httpx.AsyncClient → POST /v1/chat/completions              │
  │                                                                │
  │  Step 5: 解析响应                                              │
  │    JSON 提取 → AIResult { description, tags, scene, prompt }  │
  │                                                                │
  │  Step 6: 持久化                                                │
  │    写入 analysis_results 表 → 返回给前端                        │
  └────────────────────────────────────────────────────────────────┘
```

### 6.2 五种分析模式

| 模式 | 键值 | Prompt 策略 | 返回格式 |
|------|------|-------------|----------|
| 全部分析 | `all` | 结构化 JSON Prompt | `{description, tags, scene, prompt}` |
| 图片描述 | `describe` | 开放式中文描述 | 纯文本 |
| 关键词标签 | `tags` | 数组约束 | `["标签1","标签2",...]` |
| 场景识别 | `scene` | 场景+氛围+光线 | 纯文本 |
| Prompt 反推 | `prompt` | 生成类 prompt 反推 | 纯文本 |

## 七、安全设计

| 机制 | 实现 |
|------|------|
| **密码存储** | bcrypt 加盐哈希，不可逆 |
| **身份认证** | JWT（HS256），24 小时过期，无状态 |
| **接口鉴权** | 所有业务 API 通过 `Depends(get_current_user)` 校验 |
| **数据隔离** | 所有查询带 `user_id` 过滤，用户 A 看不到用户 B 的数据 |
| **API Key 保护** | 存入数据库，前端不回显原值（编辑时留空） |
| **文件上传安全** | 仅允许 JPG/PNG/GIF/WEBP 格式，50MB 上限 |

## 八、设计亮点（答辩重点）

### 8.1 策略模式（Provider 抽象）

```python
class BaseProvider(ABC):
    @abstractmethod
    async def analyze(self, image_base64, analysis_type) -> AIResult: ...

class CloudProvider(BaseProvider):   # OpenAI 兼容平台
    async def analyze(self, ...): ...

# 未来扩展：
# class OllamaProvider(BaseProvider):  # 本地模型
# class GeminiProvider(BaseProvider):  # Google Gemini
```

新增平台只需实现 `analyze()` 方法，对上层调用者完全透明。

### 8.2 多平台兼容

- URL 自动补全 `/v1/chat/completions`，兼容只给域名的平台
- 支持 **Bearer Token** 和 **API-Key** 两种鉴权模式
- 用户可自定义 Header 名称（如 `X-API-Key`、`x-goog-api-key` 等）

### 8.3 Prompt 工程

- 根据 `analysis_type` 动态构造不同的 System Prompt
- 要求模型以 JSON 格式返回，便于前端结构化渲染
- 容错设计：JSON 解析失败时降级为纯文本展示

### 8.4 异步非阻塞

- FastAPI 原生 async/await
- httpx.AsyncClient 异步 HTTP 请求
- 上传文件使用 aiofiles 异步 IO
- 高并发场景不阻塞主线程

### 8.5 自动 API 文档

FastAPI 自动生成交互式 Swagger UI，访问 `/docs` 即可在线调试所有接口。

## 十、答辩话术参考

> "本项目后端采用 **FastAPI + SQLAlchemy + SQLite** 的轻量架构。
>
> 核心设计思想是**策略模式**——通过 `BaseProvider` 抽象基类定义统一的 `analyze()` 接口，`CloudProvider` 作为具体实现对接 OpenAI 兼容 API。这样做的好处是：未来想接入本地模型（如 Ollama）或 Google Gemini，只需新增一个 Provider 类，现有代码零改动。
>
> 安全方面，采用 **bcrypt 加盐哈希**存储密码、**JWT 无状态认证**管理会话，所有 API 通过依赖注入校验身份，数据库查询带 `user_id` 过滤实现多用户数据隔离。
>
> AI 分析支持 **5 种分析维度**——全面分析、图片描述、关键词标签、场景识别、Prompt 反推。底层通过**结构化 Prompt 工程**让大模型返回 JSON，前端直接渲染，无需二次解析。
>
> 另外，支持用户**自定义 API 配置**——可以填写自己的 API Key、选择 Bearer Token 或 API-Key 鉴权模式，适配硅基流动、智谱、通义千问、DeepSeek 等不同平台。"

## 十一、从网页操作看后端到底做了什么

这一部分更适合答辩时讲，按照用户在页面上的实际动作来解释后端逻辑。

### 11.1 用户注册

**前端动作**：在注册页面输入用户名和密码，点击“注册”。

**后端做的事**：

1. 前端把用户名和密码通过 `POST /api/auth/register` 发给后端。
2. 后端在 `routers/auth.py` 里接收 `UserRegister` 数据。
3. 先去数据库查用户名是否已存在。
4. 如果不存在，调用 `auth.py` 里的 `hash_password()`：
   - 使用 `bcrypt` 给密码加盐哈希
   - 不保存明文密码
5. 创建 `User` 记录，写入 `users` 表。
6. 返回注册成功后的用户信息。

**用到的技术**：
- FastAPI 路由
- Pydantic 参数校验
- SQLAlchemy ORM
- bcrypt 密码哈希

---

### 11.2 用户登录

**前端动作**：输入用户名、密码，点击“登录”。

**后端做的事**：

1. 前端把登录表单提交到 `POST /api/auth/login`。
2. 这里用的是 `OAuth2PasswordRequestForm`，也就是**表单格式**，不是 JSON。
3. 后端根据用户名查数据库中的用户。
4. 使用 `verify_password()` 用 `bcrypt` 校验密码是否正确。
5. 如果正确，调用 `create_access_token()`：
   - 使用 `python-jose`
   - 生成 JWT Token
   - 设置过期时间 24 小时
6. 前端保存 token，后续请求都带 `Authorization: Bearer {token}`。

**用到的技术**：
- `OAuth2PasswordRequestForm`
- `python-jose` JWT
- `bcrypt`
- FastAPI 依赖注入

---

### 11.3 上传图片

**前端动作**：用户从手机或电脑选择一张图片，点击上传。

**后端做的事**：

1. 前端把图片作为 `multipart/form-data` 发到 `POST /api/images`。
2. 后端在 `routers/images.py` 接收 `UploadFile`。
3. 先做类型校验，只允许：
   - JPG
   - PNG
   - GIF
   - WEBP
4. 再做大小校验，限制最大 50MB。
5. 后端生成唯一文件名，避免重名覆盖。
6. 把图片保存到 `data/uploads/`。
7. 在 `images` 表里插入一条记录，保存：
   - 原文件名
   - 实际保存路径
   - 文件大小
   - 所属用户
8. 返回图片的可访问地址。

**用到的技术**：
- `UploadFile` + `multipart/form-data`
- `aiofiles` 异步写文件
- SQLite / SQLAlchemy
- Nginx 上传限制 `client_max_body_size`

**答辩时可以说**：
> “上传图片时，后端先校验格式和大小，再异步保存到本地磁盘，同时把图片元信息写入数据库，保证文件和数据记录一致。”

---

### 11.4 点击“分析”按钮

**前端动作**：用户选中一张图片，选择分析类型，点击“分析”。

**后端做的事**：

1. 前端调用 `POST /api/analysis`，把 `image_id` 和 `analysis_type` 传给后端。
2. 后端先通过 `get_current_user()` 检查当前登录用户身份。
3. 进入 `ai_service.run_analysis()`：
   - 根据 `image_id` 读取图片记录
   - 打开图片文件
   - 转成 base64
4. 查 `ai_configs` 表，获取用户自己的 AI 配置：
   - 如果指定了 `ai_config_id`，优先用它
   - 否则用默认配置 `is_default=1`
   - 如果没有用户配置，就回退到 `.env` 默认值
5. 根据配置决定鉴权方式：
   - `bearer`：`Authorization: Bearer xxx`
   - `api-key`：自定义 header，例如 `X-API-Key: xxx`
6. 通过 `httpx.AsyncClient` 请求外部 OpenAI 兼容接口。
7. 模型返回 JSON 或文本后，后端解析成统一结构 `AIResult`。
8. 把分析结果写入 `analysis_results` 表，记录模型名、耗时、分析类型、结果内容。
9. 把结果返回前端渲染。

**用到的技术**：
- FastAPI 异步路由
- Base64 图片传输
- httpx 异步请求
- Prompt 工程
- JSON 解析
- ORM 持久化

**答辩时可以说**：
> “点击分析后，后端会先把图片转成 base64，再根据用户配置的 API Key 调用外部大模型接口，最后把分析结果标准化保存到数据库。”

---

### 11.5 AI 配置页面

**前端动作**：用户进入“AI 配置”页面，新建或修改自己的 API 信息。

**后端做的事**：

1. 前端提交 `base_url`、`api_key`、`model_name`、`auth_mode`、`auth_header` 等字段。
2. 后端在 `routers/ai_configs.py` 中接收这些配置。
3. 保存到 `ai_configs` 表。
4. 如果勾选“设为默认配置”，先把该用户其他配置的 `is_default` 清零。
5. 编辑配置时，`api_key` 不会回显，只在用户重新填写时才更新。

**用到的技术**：
- Pydantic 请求模型
- SQLAlchemy ORM
- 用户级配置隔离
- 前后端字段映射

---

### 11.6 查看图片列表 / 分析历史

**前端动作**：打开首页或历史记录页，查看上传图片和分析记录。

**后端做的事**：

1. 通过 `GET /api/images` 或 `GET /api/analysis` 获取数据。
2. 查询时都按 `user_id` 过滤，确保只能看到自己的内容。
3. 把结果序列化返回给前端。
4. 图片地址通过 `/uploads/文件名` 形式公开访问，由 Nginx 反代到后端静态文件目录。

---

### 11.7 删除图片或配置

**前端动作**：点击删除按钮。

**后端做的事**：

1. 接收删除请求。
2. 校验资源是否属于当前用户。
3. 删除数据库记录。
4. 图片删除时，通常同时清理磁盘文件。
5. 返回删除成功。
